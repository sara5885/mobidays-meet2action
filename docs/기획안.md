# 회의록 자동 정리 · 액션아이템 추출 시스템 — 기획안

> 모비데이즈 AI Tech Lab 사전과제 · 지원자 황수진
---

## 1. 문제 재정의 — 무엇을 우선 해결하는가

모비데이즈는 매주 광고주별 캠페인 회의를 운영하고, 회의 후 담당자가 회의록을 정리해
액션아이템을 트래킹 시스템에 옮긴다. 여기서 두 가지 페인포인트가 반복된다.

1. **수기 정리 부담** — 회의록 + 액션아이템 정리에 회의당 30~60분 소요.
2. **액션아이템 누락** — 한국어 회의 특성상 결정이 흐릿하게 끝나거나("일단 두고 봐요"),
   R&R이 암묵적으로 처리되어("그건 제가 챙길게요") 책임자/기한이 명시되지 않은 채 흘러간다.

**우선 해결 대상은 2번(누락)이다.** 1번(시간)은 자동화로 비교적 쉽게 줄지만, 2번은
"무엇이 액션아이템인지, 누가 책임자인지"를 판단해야 하는 문제라 난이도가 높고 비즈니스
임팩트가 크다(누락된 액션은 캠페인 일정 지연·광고주 신뢰 하락으로 직결). 따라서 본 시스템은
**"흐릿한 한국어 회의에서 액션아이템을 누락 없이, 그러나 신뢰할 수 있게 구조화"**하는 데
설계의 무게를 둔다. 핵심 난제는 *신뢰할 수 없는 LLM 출력을 신뢰 가능한 데이터 자산으로
만드는 것*이다.

## 2. 시스템 아키텍처

```
[수집]            [처리·저장]              [AI 추출/정리]              [분배]          [분석]
음성(mp3) ─►Whisper STT─┐                ┌─► 액션아이템(검증·재시도·환각필터·confidence) ─┬─► Slack 페이로드
                       ├►정제─►청크─►DuckDB ┤                                          │
transcript JSON ───────┘ (화자정규화·약어   └─► 회의록(요약·안건·결정사항)                 ├─► Streamlit 대시보드
                          사전·잡음제거)        [LLM: mock/Gemini/Ollama]                │   · 📄 회의록(목록→상세)
  + 참석자 명단(employees/meeting_participants)                                          │   · 위젯4 + 상태/담당자 편집
                                            action_status / status_history ◄────────────┘   (사람이 관리, 재실행 보존)
```

### 단계별 도구·모델 선택과 trade-off

| 단계 | 선택 | 대안 | 선택 근거 (trade-off) |
|---|---|---|---|
| STT | **로컬 Whisper** + 제공 transcript | 외부 STT API | **외부 유출 금지** 제약 → 클라우드 STT 불가. 로컬 Whisper로 mp3 직접 처리(비용 0·온프레미스). 제공 transcript로도 검증 |
| 화자 구분 | **텍스트+명단 기반 LLM 매핑** | pyannote/whisperx 음성 diarization | 음성 diarization이 더 정확하나 GPU·외부 의존이 무겁고, 그래도 이름·역할은 못 줌. 비용 0 + 명단 활용으로 절충 (아래 3단 계층) |
| 저장소 | **DuckDB** | SQLite / Postgres | 파일 1개 임베디드 + 컬럼형 OLAP라 대시보드 집계에 강함. 100명 PoC엔 Postgres 운영비가 과함. SQLite보다 분석 쿼리 우수 |
| LLM | **mock / Gemini 2.5 Flash / Ollama** | OpenAI | provider 추상화로 3종 토글. Gemini=무료 native JSON, **Ollama=무료·무제한·온프레미스로 외부 유출 금지 제약을 실제로 충족**, mock=결정적 시연 |
| 검증 | **pydantic** | 수동 파싱 | LLM 출력을 타입·범위·필수값으로 강제 → 적재 전 차단 |
| 대시보드 | **Streamlit** | Metabase/Superset | Python 단일 스택, 빠른 PoC, DuckDB 직접 연결 |
| 키워드 | **BoW(TF-IDF류)** | 임베딩 클러스터링 | 회의 수 적은 PoC엔 BoW가 안정적·설명가능. 데이터 누적 시 임베딩으로 고도화 |

**모듈 분리 기준**: I/O(loader·db) · 순수 변환(preprocess) · LLM 경계(llm·extract) · 표현(dashboard)을
분리. LLM provider 교체나 스키마 변경이 다른 층에 번지지 않게 했다.

### 화자 구분(diarization) 3단 계층

Whisper는 받아쓰기만 하고 "누가 말했나"는 안 준다. 화자 구분은 정확도순으로 3단계이며 **경쟁이 아니라 보완**이다.

| 계층 | 방법 | 강점 / 한계 | 본 PoC |
|---|---|---|---|
| 1 | 음성 diarization (pyannote/whisperx) | 목소리로 화자 전환을 정확히 분리 / GPU·외부 의존 무거움, 결과는 `SPEAKER_00` 뿐(이름·역할 모름) | 향후 확장 |
| 2 | 텍스트 + 참석자 명단 기반 LLM 매핑 | 비용 0, 명단을 닫힌 후보로 줘서 owner 정확도↑ / 음성 정보 없음 | **구현** |
| 3 | 사람이 대시보드에서 수정 | 최종 안전망(틀린 매핑·미상 화자 보정), 재실행에도 보존 | **구현** |

핵심: 음성을 붙여도(1단계) `SPEAKER_00`→"재현" 매핑엔 결국 **명단(2단계)과 사람 보정(3단계)이 필요**하다.
그래서 본 PoC는 비용 0인 2·3단계를 구현하고, 1단계는 음성이 있을 때 붙이는 향후 확장으로 둔다.

## 3. 데이터 스키마 설계 근거

DuckDB에 9개 테이블. **정규화로 중복을 제거하되, 추적성·조회 성능을 위해 선택적으로 비정규화**했다.

| 테이블 | 역할 | 정규화 결정 |
|---|---|---|
| `meetings` | 회의 메타(광고주·날짜) | 정규화. 광고주명은 여기 한 곳에만 저장 |
| `utterances` | 화자분리 발화 원문 | 정규화. 원문 보존(추적성), 정제는 청크 시점에만 적용 |
| `chunks` | LLM 입력 단위 | 정규화 |
| `meeting_summary` | 회의록(요약·안건·결정사항) | 재실행마다 갱신. agenda/decisions는 JSON 배열 |
| `action_items` | AI 추출 결과 | `source_quote`·`source_seg_ids`를 **비정규화**로 함께 저장 → 조인 없이 근거 추적 |
| `action_status` | 진행상황(사람 소유) | 추출과 **분리** (재실행에도 보존). owner_override로 담당자 수정도 보존 |
| `status_history` | 상태 변경 이력 | 감사 추적 |
| `employees` | 직원 마스터(id·이름·역할) | **정규화**: 직원 정보는 여기 한 곳에만. `is_adhoc`로 STT 미상 화자 구분 |
| `meeting_participants` | 회의↔직원 연결 | **FK만** 저장(다대다). 역할 변경 시 employees 한 곳만 수정 → 갱신 이상 방지 |

### 추출/트래킹 레이어 분리 (핵심 결정)

LLM 출력은 비결정적이라 재추출 시 제목이 미세하게 흔들릴 수 있다. 그래서
**회의(meeting_id)를 멱등 단위로 삼아 `action_items`를 통째 교체(delete-replace)**하되,
사람이 바꾼 진행상황은 안정 식별자 `action_key`(= meeting_id + 정규화 제목 해시)로
`action_status`에 따로 보존한다. 결과적으로 **"AI 추출은 멱등하게 재생성 + 사람의 상태 관리는 생존"**이
동시에 성립한다. (멱등성 요구사항 + 진행상황 업데이트 루프를 한 설계로 해결)

### 액션아이템 필드 정의 근거

| 필드 | 근거 |
|---|---|
| `title` | 액션 한 줄 요약 (트래킹 시스템 이관 단위) |
| `owner_role` | 암묵적 R&R 매핑 결과. 불명확하면 `null` (억지 배정 금지) |
| `due` | 기한 텍스트. "다음주 금요일" 같은 자연어 보존 |
| `status` | open/in_progress/done/**blocked**(지연+사유) |
| `confidence` | 추출 신뢰도 0~1. **낮은 항목만 사람이 검수**하는 운영 흐름의 핵심 |
| `source_seg_ids`·`source_quote` | 근거 발화 인용. **환각 방지·추적성**. 입력에 없는 seg_id 인용 시 confidence 하향 |

## 4. Before / After 임팩트 추정 (사내 100명 기준)

**가정**: 광고주 약 30곳, 주 1회 회의 → **주 30건 회의**. 회의당 수기 정리 평균 45분.

| 지표 | Before(수기) | After(본 시스템) | 근거 |
|---|---|---|---|
| 회의당 정리 시간 | 45분 | **10분** (자동초안 + 검수만) | 정리·이관을 자동화, 사람은 낮은 confidence 항목만 확인 |
| 주간 절감 시간 | — | **30건 × 35분 ≈ 17.5시간/주** | 연 환산 ≈ 약 900시간 |
| 액션아이템 누락률 | ~20% | **~5%** | 흐릿한 R&R도 추출 후 confidence로 표시 → 검수에서 회수 |
| 사람 검수 범위 | 100% | **~30%** | confidence < 0.6 항목만 검수 (분포상 약 30%) |

*추정 한계*: 수치는 가정 기반이며, 도입 후 실제 회의 로그로 보정 필요(4주 운영·KPI 측정 권장).
핵심은 절대값이 아니라 **"검수 범위를 70% 줄이면서 누락을 잡는다"**는 구조적 개선이다.

## 5. 실패 시나리오 & 대응

**① LLM이 환각하거나 스키마를 위반한다**
- 위험: 존재하지 않는 액션아이템 생성, 잘못된 JSON, 입력에 없는 발화 인용.
- 대응: (a) pydantic으로 타입·필수값·범위 강제 → 위반 시 **강화 프롬프트로 재시도**(MAX_LLM_RETRIES).
  (b) `source_seg_ids`가 실제 입력 발화에 존재하는지 **환각 필터** → 없으면 제거 + confidence 하향.
  (c) 재시도 소진 시 빈 결과 + 로그(파이프라인은 죽지 않음).

**② STT 받아쓰기 오류 / 화자 오분리 / 약어 오인식**
- 위험: 받아쓰기 자체가 틀림. **실측**(Whisper `small`)에서 "채널별 성과→채널된 손가락",
  "GA→GAS", "이중 집계→이중 집게", "제안서→제한서" 등 한국어·광고 약어 오인식 다수 확인
  (`docs/STT_품질검증.md`). STT 품질이 **전체 파이프라인 품질의 상한**이다.
- 대응: (a) **고성능 STT가 토대** — 큰 Whisper(`medium`/`large`)·도메인 튜닝·온프레미스 STT.
  (b) STT는 틀린다는 전제로 **약어 사전 주입 + LLM 화자 매핑 + confidence·사람 검수**로 완충.

**③ 외부 API 장애 / 무료 quota 초과**
- 위험: Gemini 429/장애로 추출 중단.
- 대응: **429 백오프**(권장 대기 후 재시도) + **provider 추상화**로 로컬 Ollama/mock 전환 가능.
  멱등 설계라 **재실행 시 중복 없이 이어서 처리**. 실서비스는 사내 LLM(Ollama 등 온프레미스)으로
  대체해 외부 유출 금지까지 동시에 충족 — 본 PoC에서 Ollama로 실제 구현·검증함.

## 6. 도입 후 4주 운영·검증 계획 (KPI·모니터링)

PoC를 사내에 도입한 뒤 "효과가 있는가, 믿고 쓸 수 있는가"를 숫자로 검증하는 계획.

> **현재 측정치 (PoC, gold=수기 9+12건 기준)**: 로컬 Ollama(qwen2.5:7b) 기준
> **Precision 0.81 · Recall 0.81 · F1 0.81 · 담당자 정확도 0.77** (`make eval`).
> gold가 저자 1인 기준이라 절대값보다 *모델·프롬프트 변경 시 상대 비교*에 의미가 있다.
> (타입 보정 추가 전 F1 0.67 → 추가 후 0.81로, 검증 로직 개선 효과를 수치로 확인)

### KPI (성공 지표)

| 지표 | 정의 | 목표 | 현재(PoC) | 측정 방법 |
|---|---|---|---|---|
| Precision / Recall | AI 추출 액션아이템의 정확도/누락 | F1 ≥ 0.8 | **F1 0.81** | gold(수기) 대비 `scripts/eval_extraction.py` |
| 누락률 | 회의당 놓친 액션아이템 비율 | 20% → ≤ 5% | (R=0.81 → 누락 ~19%) | 1주차 수기 vs AI 병행 비교 |
| 정리 시간 | 회의당 회의록·액션 정리 소요 | 45분 → ≤ 10분 | — | 담당자 타임로그 |
| 채택률 | AI 초안을 그대로/수정 사용한 비율 | ≥ 70% | — | 상태 편집·수정 로그(status_history) |
| confidence 신뢰성 | 高 confidence가 실제로 맞는 비율(calibration) | 0.9+ 구간 정확도 ≥ 0.9 | — | 검수 결과와 confidence 교차 분석 |

### 모니터링 (운영 중 상시 관찰)

- LLM 실패율: 스키마 위반·재시도·429 빈도 (급증 시 모델/입력 품질 경보)
- confidence 분포 추이: 평균이 떨어지면 입력(STT) 품질 저하 신호
- 사람 수정 빈도: 특정 회의·담당자에서 수정이 잦으면 프롬프트 튜닝 트리거
- 처리 지연·비용(토큰). 대시보드의 confidence 분포·완료율 위젯이 모니터링 기반.

### 4주 타임라인

| 주차 | 활동 | 게이트(통과 기준) |
|---|---|---|
| 1주 | 소수 팀 파일럿, 수기 vs AI **병행 비교**로 기준선(P/R) 측정 | F1 측정값 확보 |
| 2주 | 약어 사전·프롬프트 튜닝, 대상 팀 확대 | F1 개선 추세 |
| 3주 | 전사 확대, 모니터링 대시보드 상시 운영 | 채택률 ≥ 50% |
| 4주 | KPI 종합 평가 → 확대/롤백 결정 | F1 ≥ 0.8 & 정리시간 목표 달성 |

> 1주차 "기준선 측정"은 본 PoC의 `scripts/eval_extraction.py`(gold vs LLM)를 그대로 사용한다.

## 부록 · 구현한 가산점 & 향후 확장

**이미 구현한 가산점 / 추가 구현**
- **회의록 자동 정리** — 요약·안건·결정사항을 LLM이 추출(흐릿한 논의는 결정에서 제외),
  대시보드에서 회의별 목록→상세로 열람 + 원문 transcript 대조 검증
- 실제 LLM 호출 PoC — Gemini 2.5 Flash + 로컬 Ollama (provider 추상화로 토글)
- 로컬 Whisper STT + 화자 분리 — Whisper JSON에 화자 채워 넣기(id-매핑), 임의 음성/텍스트 입력
- 직원 마스터 정규화 — employees + meeting_participants(FK), is_adhoc로 미상 화자 격리
