# 코드 가이드 — 폴더·파일 전체 해설

이 문서는 리포 전체를 **폴더 → 파일 → 함수** 순(top-down)으로, "무엇을/왜"까지 설명합니다.
면접 대비 + 내일 작업 재개용.

---

## 0. 폴더 구조 한눈에

```
mobidays-meet2action/
├── README.md                 # 실행법·스택근거·아키텍처 (제출 필수)
├── AI_USAGE.md               # AI 활용 내역 (제출 필수)
├── Makefile                  # 한 줄 명령어 모음 (make run, test, demo …)
├── requirements.txt          # 파이썬 의존성
├── .env.example              # 환경설정 템플릿 (LLM provider 등)
├── .gitignore                # git 제외 목록 (.env, .venv, db 등)
│
├── src/meeting_ai/           # ★ 핵심 패키지 (파이프라인 본체)
│   ├── config.py             # 모든 설정·경로 중앙관리
│   ├── schemas.py            # pydantic 데이터 모델(계약서)
│   ├── transcript_loader.py  # transcript JSON → Utterance
│   ├── preprocess.py         # 정제 + 약어사전 + 청크 분리
│   ├── db.py                 # DuckDB 적재 + 멱등성 + 상태 트래킹
│   ├── prompts.py            # LLM 프롬프트 설계
│   ├── extract.py            # LLM 호출 + 검증/재시도/환각필터
│   ├── keywords.py           # BoW 반복 이슈 키워드
│   ├── slack_payload.py      # Slack 메시지 JSON 생성
│   └── llm/                  # LLM provider 추상화
│       ├── base.py           # 인터페이스(추상 클래스)
│       ├── mock.py           # 가짜 응답(정답지=gold)
│       ├── gemini.py         # 실제 Gemini API
│       └── ollama.py         # 로컬 LLM (무료·무제한)
│
├── dashboard/app.py          # Streamlit 대시보드(위젯4 + 상태편집)
├── scripts/                  # 독립 실행 스크립트
│   ├── gen_synthetic.py      # 합성 회의 3건 생성
│   ├── simulate_progress.py  # 진행상황(상태) 시뮬레이션
│   ├── check_gemini.py       # Gemini 연결 점검
│   ├── run_local_stt.py      # Whisper STT + 화자분리
│   └── eval_extraction.py    # precision/recall 평가
│
├── tests/test_pipeline.py    # 멱등성·스키마·상태보존 테스트
├── data/
│   ├── raw/                  # 입력 transcript JSON + mp3
│   ├── stt/                  # Whisper STT 출력
│   └── db/                   # DuckDB 파일(.gitignore)
└── docs/                     # 기획안·녹화시나리오·이 가이드
```

**왜 이렇게 나눴나 (레이어 분리)**: 데이터가 흐르는 순서대로 모듈을 쪼갰습니다.
I/O(loader·db) · 순수 변환(preprocess·keywords) · LLM 경계(llm·prompts·extract) · 표현(dashboard).
한 층을 바꿔도(예: Gemini→Ollama, 스키마 변경) 다른 층이 안 깨지게 하기 위함입니다.
`src/` 아래 패키지로 둔 건 import 경로를 깔끔히 하고 스크립트/대시보드/테스트가 공통으로 쓰기 위함.

## 1. src/meeting_ai — 핵심 패키지

### config.py — 설정 중앙관리
모든 경로(ROOT, DATA_DIR, DB_PATH)와 환경변수(LLM_PROVIDER, GEMINI/OLLAMA 설정,
MAX_LLM_RETRIES, WHISPER_MODEL)를 한 곳에서 읽습니다. `.env`를 `load_dotenv()`로 로드.
- **왜**: 설정이 코드 여기저기 흩어지면 바꾸기 어렵고 실수가 남. 한 파일로 모으면
  "provider 바꾸기 = .env 한 줄"이 됨.

### schemas.py — 데이터 '계약서' (pydantic)
파이프라인 전 구간이 주고받는 데이터의 **형태와 규칙**을 정의.
- `Utterance`: 발화 1개(화자·역할·시간·텍스트)
- `Chunk`: LLM 입력 단위(발화 묶음)
- `ActionItem`: 액션아이템(title, owner_role, due, status, confidence, source_seg_ids/quote)
- `ActionStatus`: open/in_progress/done/blocked
- **검증기(validator)**: title 비면 거부 / confidence는 0~1 / **타입 보정**(source_quote가
  리스트로 오면 문자열로, seg_ids "1,2"→[1,2], 빈 owner→None)
- **왜**: LLM 출력은 못 믿음. 이 '계약서'를 통과해야만 데이터로 인정 → 깨진 데이터 차단.
  보정 검증기는 "사소한 형식 오류는 거부 말고 고쳐서 살린다"는 철학(F1 0.67→0.81 기여).

### transcript_loader.py — 입력 읽기
transcript JSON을 읽어 `Utterance` 리스트로 변환. 두 포맷을 모두 흡수(방어적 파싱):
제공 실데이터(speaker=이름, role 내장, 타임스탬프 없음, speakers 리스트)와
일반 STT(speakers={코드:역할}, 타임스탬프).
- 핵심: `_build_speaker_role_map`(화자→역할 맵), 세그먼트별 역할 우선순위(내장 role → 맵 → 코드)
- **왜**: 데이터 포맷이 확정 안 됐을 때도, Whisper 출력으로 바꿔도 같은 로더로 처리되게.

### preprocess.py — 정제 + 청크
1) `clean_text`: 머뭇거림(음·어·그…) 제거, 말줄임표 정리
2) `dedup_consecutive`: 같은 화자의 연속 중복 발화 제거
3) `ABBREV`/`abbrev_glossary`: 광고 약어 사전 + 이 회의에 등장한 약어만 추출
4) `to_chunks`: 화자 라벨 붙여 의미 단위 청크로 묶기(짧은 회의=1청크)
- **왜**: 원문(utterances)은 보존하고 정제는 청크 만들 때만 적용 → 추적성 유지.
  약어 사전을 회의별로 추려 프롬프트에 주입 → 불필요한 약어로 프롬프트 오염 방지.

### db.py — 저장 + 멱등성 + 상태 트래킹 (설계 핵심)
DuckDB에 6개 테이블. 두 레이어로 분리:
- **추출 레이어**: meetings/utterances/chunks/action_items — 재실행마다 `meeting_id` 단위
  delete-replace(**멱등성**). `make_action_key`(meeting_id+제목 해시)로 안정 식별자 부여.
- **트래킹 레이어**: action_status/status_history — 사람이 바꾼 상태. 파이프라인이 안 건드림.
- `upsert_action_items`: 추출 결과 교체 + 신규 action_key만 상태행 초기화(기존 사람 수정분 보존)
- `update_status`: 상태 변경 + 이력 기록(blocked면 지연사유)
- **왜**: LLM 재추출 시 제목이 흔들려도 회의 단위로 통째 교체하면 멱등. 동시에 사람이
  바꾼 상태는 action_key로 따로 보존 → "추출은 멱등 재생성 + 상태는 생존" 양립.

### prompts.py — LLM 지시문
- `SYSTEM`: role 지정 + 도메인 지식(약어, 암묵 R&R 규칙) + 추출 규칙
- `OUTPUT_SCHEMA_HINT`: 출력 JSON 스키마 명시
- `FEWSHOT`: "그건 제가 챙길게요"→담당자 매핑 예시 1개
- `build_user_prompt`: 약어 용어집 + few-shot + 스키마 + 회의 발화를 조립
- **왜**: 역할·배경·시범(few-shot)을 줄수록 LLM이 도메인에 맞게 답함. 발화에 `#seg_id [역할]`
  라벨을 붙여 "누가 말했나"를 알 수 있게 → 암묵 담당자 매핑 가능.

### extract.py — LLM 호출 + 신뢰성 엔지니어링 (평가 핵심)
- `extract_from_chunk`: 프롬프트 생성 → provider 호출 → 파싱·검증 → 실패 시 재시도
- `_complete_with_backoff`: 429(rate limit) 시 권장 시간 대기 후 재시도(스키마 재시도와 별개)
- `_parse_and_validate`: 코드펜스 제거 + JSON 파싱 + pydantic 검증
- `_postprocess`: **환각 필터**(없는 seg_id 인용 제거 + confidence 강등) + 담당자 없으면 상한
- **왜**: "재시도(형식 깨짐)·환각필터(근거 지어냄)·백오프(API 한도)·confidence 보정"으로
  신뢰 불가 출력을 신뢰 가능한 데이터로. = 평가 포인트 그 자체.

### keywords.py — 반복 이슈 키워드(BoW)
- `tokenize`: 정규식 토큰화 + 조사 제거(`_strip_josa`) + 불용어/화자이름 필터
- `top_keywords`: 빈도(tf) × 등장 회의 수(df) 점수로 정렬 → 여러 회의 반복 키워드 강조
- **왜**: 형태소 분석기 설치 부담 없이 가볍게. df 가중으로 "일회성"이 아닌 "구조적 이슈"를 부각.
  데이터 적은 PoC엔 임베딩 클러스터링보다 안정·설명가능.

### slack_payload.py — 분배 산출물
액션아이템을 Slack 메시지 블록(JSON)으로 변환. confidence<0.6엔 "⚠️확인필요" 플래그.
- **왜**: 실무에선 회의 후 Slack으로 할 일 자동 공유. 그 양식 샘플 제공(과제 산출물).

### llm/ — provider 추상화
- `base.py`: `LLMProvider` 추상 클래스(`complete_json(system, user)` 인터페이스)
- `mock.py`: 회의별 정답 픽스처(사람이 작성) — 시연용 + **eval의 gold(정답지)**
- `gemini.py`: 실제 Gemini API(JSON 강제, system_instruction)
- `ollama.py`: 로컬 LLM(urllib로 localhost:11434 호출, 무료·무제한·온프레미스)
- `__init__.get_provider()`: config.LLM_PROVIDER 보고 구현체 선택
- **왜**: 호출부(extract)는 "JSON 문자열 받는다"만 알면 됨. 모델 교체가 다른 코드에 안 번짐.
  외부 유출 금지(ollama)·rate limit·시연(mock)을 한 인터페이스로 해결.

## 2. dashboard/app.py — Streamlit 대시보드
DuckDB를 읽어 위젯 4개 + 상태편집 UI 렌더.
- 상단 KPI 카드(델타 포함) → 위젯1 추이(막대+완료율선, 최근 12주 고정축) →
  위젯2 담당자별 미완료 → 위젯3 키워드 칩 → 위젯4 confidence 분포+드릴다운 →
  상태 편집 폼(저장 시 `db.update_status` 호출)
- 상태는 `action_items`와 `action_status`를 join해서 읽음
- **왜**: "차트 나열"이 아니라 의사결정 흐름(추세→병목→반복과제→검수). custom CSS로 카드화,
  `st.container(border=True)`로 위젯을 박스 안에 담음.

## 3. scripts/ — 독립 실행 도구
- `gen_synthetic.py`: 제공 데이터와 같은 구조의 가상 회의 3건 생성(추이/키워드 위젯용,
  '픽셀/전환' 이슈 반복 설계). **왜**: 실제 회의가 1건뿐이라 추이가 의미를 가지려면 여러 건 필요.
- `simulate_progress.py`: 회의 경과시간 기준으로 상태(done/in_progress/blocked) 부여 →
  완료율 추이가 의미 있게. (가산점 '진행상황 업데이트 루프'의 mock)
- `check_gemini.py`: Gemini 키/모델/JSON출력 점검(`make gemini-check`)
- `run_local_stt.py`: mp3 → Whisper 받아쓰기 → LLM 화자분리 → transcript JSON. 임의 파일 지원.
- `eval_extraction.py`: gold(mock) vs 실제 LLM 추출을 매칭해 P/R/F1·담당자정확도 산출.
- **왜 scripts로 분리**: 파이프라인 본체(src)와 달리 "가끔 한 번 돌리는 보조 작업"이라 분리.

## 4. tests/test_pipeline.py
- `test_run_and_idempotency`: 두 번 실행해도 액션아이템 수 동일(멱등성 증명)

- **왜**: 핵심 보장(멱등성·검증·상태보존)을 코드로 증명. 회귀 방지.

## 6. 데이터 흐름 다시 한 번 (파일 매핑)

```
data/raw/*.json
  └─ transcript_loader.py → Utterance[]
       └─ preprocess.py → Chunk[] ─────────────► db.py (utterances/chunks 적재)
            └─ extract.py (+ prompts.py + llm/*) → ActionItem[]
                 ├─ schemas.py 로 검증·보정
                 └─ db.py (action_items 적재) ──► dashboard/app.py (+ keywords.py)
                      └─ slack_payload.py → data/slack_payload_sample.json
   (사람) dashboard 상태편집 → db.update_status → action_status/status_history
```

핵심 한 줄: **"LLM은 못 믿는다"는 전제로, 잘 지시(prompts)하고 → 받은 걸 검증·보정(schemas·extract)하고
→ 멱등하게 저장(db)하고 → 의사결정용으로 보여준다(dashboard)."**
