# STT & 화자 매핑 품질 검증 (실측)

제공된 실제 음성 `ko_meeting_3speakers_4min_faster.mp3`(약 4분, 화자 3명)를 로컬 Whisper로
직접 받아쓰고, 동봉 정답지 transcript와 대조해 품질을 측정한 기록.
(LLM·mock·정답지 복사 일절 없이 순수 실측. 외부 API 미사용 = 회의 내용 외부 유출 없음.)

---

## 1. 시도 전략

STT는 학습된 모델이라 재학습 없이 품질을 올릴 수 있는 **엔지니어링 레버**를 가설로 세우고 하나씩 실험했다.
1. **모델 크기** (small ↔ medium)
2. **어휘 바이어싱** — Whisper `initial_prompt`에 도메인 용어(광고주·약어)를 줘서 디코딩 유도
3. **화자 매핑 단위** — 조각을 그대로 매핑 vs 같은 화자 조각을 병합 후 매핑

각 레버를 `--no-bias` 등으로 **변수 통제**하며 정답지 대비로 측정했다.

---

## 2. 실험 & 결과

### 2-1. 받아쓰기 정확도 (텍스트)

전체 문자 유사도(medium, 정답 대비) **94.0%** — 일상 어휘는 양호. 그러나 **도메인 고유어에서 집중적으로 오류**:

| 정답 | small | medium |
|---|---|---|
| 채널**별 성과**랑 예산 배분 | 채널된 **손가락** 대분 ❌ | 채널된 **성과**랑 대본 △ |
| 메타랑 **GA** | **GAS차** ❌ | **GA** ✅ |
| **제안서** | **제한서** ❌ | **제안서** ✅ |
| **이중 집계** / **누끼** / **채린** / **A/B** | 집게·눕기·체리·AB돌보 ❌ | 이좀집계·눕기·체리·AB돌보 ❌ |

도메인 용어 정확도(medium): 노바드림·GA·CTA·픽셀·세그먼트·제안서 ✅ / A/B(0/4)·누끼(0/3)·채린(0/2)·이중집계(0)·목업(0) ❌

- **모델 크기(small→medium)가 가장 효과적** — GA·성과·제안서를 교정.
- **어휘 바이어싱(initial_prompt)은 효과 ≈ 0** — "GA·노바드림"을 줘도 음향적으로 비슷한 구간은 그대로(`--no-bias` 대조 확인).

### 2-2. 화자 매핑 정확도 (diarization)

정답지 화자를 ground-truth로, 라벨을 가리고 LLM이 매핑하게 한 뒤 발화별 비교 (`make eval-diar`, Ollama):

| 조건 | 정확도 |
|---|---|
| 문장 단위(병합 상태) | **37.8%** (14/37) |
| 조각 단위(과분할) | 45.8% (44/96) |

- 두 방식 모두 **낮음**(3명 무작위 33% 대비 소폭). 단위가 달라 둘의 직접 비교는 큰 의미 없음.
- 오답 예: "채널별 성과랑 예산 배분 보고…"(정답 수아 → 예측 채린) — **누가 말해도 이상하지 않은 발화**는 텍스트만으로 화자 특정 불가.

---

## 3. 결론

- **받아쓰기**: 전체 94%로 양호하나 **비즈니스 핵심 용어(약어·고유명사)에서 집중적으로 틀린다.**
  단순 WER이 아니라 **도메인 용어 정확도**를 따로 봐야 한다. 효과적 레버는 **모델 크기**(>어휘 바이어싱).
- **화자 매핑**: 텍스트 기반은 **37~46%로 한계가 명확**하다. 음성 정보 없이는 정확한 화자 특정이 어렵다.
- 따라서 **"고성능 STT + 음성 기반 화자 분리"가 정확도의 토대**이며, 텍스트·LLM 단계는 그 위의 보정층이다.

---

## 4. 왜 이런 한계가 있는가

- **STT 오인식**: Whisper는 범용 음성으로 학습돼 **한국어 + 영어 약어 혼용(GA, A/B)·도메인 신조어(누끼, 키비주얼)**
  분포를 거의 못 봤다. 음향적으로 비슷한 일반어로 디코딩된다("GA"→"GAS차"). `initial_prompt`는 약한 힌트라
  음향 확률을 못 뒤집는다 → 바이어싱 무효.
- **화자 매핑**: 화자 정보는 본질적으로 **목소리(성문)**에 있는데 STT 텍스트엔 그게 없다. 게다가 회의 발화 다수가
  "누가 말해도 되는" 내용이라, 텍스트만으로는 화자를 가를 단서가 부족하다. (역할↔발화 연결이 약함)

---

## 5. 대응 (본 시스템 설계)

| 한계 | 대응 |
|---|---|
| 받아쓰기 오류 | 모델 크기↑(가장 효과적) · 온프레미스 고성능/도메인 STT(향후) |
| 약어·도메인어 | LLM이 문맥으로 보정 → **최종 산출물(액션아이템·회의록)은 깨끗** (raw는 추적용 보존) |
| 화자 매핑 한계 | ① 음성 diarization(pyannote, 향후) ② 참석자 명단으로 후보 축소 ③ **대시보드 사람 검수(현 구조의 필수 안전망)** |

> 핵심: **STT/화자매핑을 "완벽히" 만들려 하지 않는다.** 모델·음성으로 토대를 올리되, 남는 노이즈는
> LLM 문맥 보정과 사람 검수로 흡수한다. 한계를 측정해 "안 되는 레버(바이어싱)는 버리고, 되는 레버(모델 크기)와
> 보정층에 집중"한 것이 본 검증의 결론이다.

### 재현 명령
```bash
# 받아쓰기 baseline (LLM 개입 0)
PYTHONPATH=src python scripts/run_local_stt.py data/raw/ko_meeting_3speakers_4min_faster.mp3 --raw-only --no-bias
# 모델/바이어싱 비교: WHISPER_MODEL=medium, --no-bias 토글
# 화자 매핑 정확도
make eval-diar PROVIDER=ollama
```
