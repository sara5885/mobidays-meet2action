# STT 품질 검증 (실측)

제공된 실제 음성 `ko_meeting_3speakers_4min_faster.mp3`를 **로컬 Whisper(`small`)로 직접 받아쓰기**한 뒤,
동봉 정답지 transcript와 대조한 결과. (LLM·mock·정답지 복사 일절 없이 순수 STT 출력)

재현:
```bash
PYTHONPATH=src python scripts/run_local_stt.py data/raw/ko_meeting_3speakers_4min_faster.mp3 --raw-only
# → data/stt/..._raw.json (Whisper 원본만)
```

## Whisper(small) vs 정답지 — 오인식 사례

| 정답지(원본) | Whisper 받아쓰기 | 유형 |
|---|---|---|
| 채널별 성과랑 예산 배분 | 채널된 손가락 예산 대분 | 구절 통째 오인식 |
| 메타랑 **GA** 숫자가 안 맞아요 | 메타랑 **GAS차**가 안 맞아요 | 약어(GA) 오인식 |
| **이중 집계** | 이중 **집게** | 동음이의 오인식 |
| **제안서** | **제한서** | 반복 오인식 |
| 랜딩은 아직 손 못 **댔어요** | 손 못 **뗐어요** | 미세 오류 |
| 노바드림 / 픽셀 / 랜딩 / 타겟 세그먼트 | (정확) | 고유명사 일부 정상 |

→ 출력이 정답지와 **부분적으로 다르다**는 점이 곧 "정답지를 베끼지 않은 진짜 STT 출력"이라는 증거다.

## 실험: 모델 크기 vs 어휘 바이어싱 (어느 레버가 효과적인가)

같은 음성에 대해 small/medium × 바이어싱(initial_prompt) on/off로 받아쓰기 비교.

| 정답 | small | medium | 바이어싱(medium on/off) |
|---|---|---|---|
| 채널별 **성과**랑 예산 | 채널된 **손가락** 대분 ❌ | 채널된 **성과**랑 대본 △ | 차이 없음 |
| 메타랑 **GA** 숫자 | **GAS차** ❌ | **GA** ✅ | 차이 없음 |
| **제안서** | **제한서** ❌ | **제안서** ✅ | 차이 없음 |
| **이중 집계** | 이중 집게 ❌ | 이좀 집계 ❌ | 차이 없음 |
| **누끼** 컷 / **채린** / **A/B** | 눕기·체리·AB돌보 ❌ | 눕기·체리·AB돌보 ❌ | 차이 없음 |

**발견:**
- **모델 크기(small→medium)가 가장 효과적인 레버** — GA·성과·제안서를 바로잡음.
- **initial_prompt 어휘 바이어싱은 효과 ≈ 0** — "GA·노바드림"을 프롬프트에 줘도 음향적으로 비슷한
  구간(이중 집계, AB 돌고 등)은 교정 못 함. (`--no-bias`로 baseline 대조 확인)
- 고유명사·약어·동음이의는 모델 크기로도 한계가 남음.

## 정량 결과 (medium, 정답지 대비)

실제 mp3를 Whisper `medium`으로 받아쓴 뒤 동봉 정답지와 대조:

- **전체 문자 유사도 94.0%** (정답 2,523자 / STT 2,393자) — 일상 어휘 받아쓰기는 양호.
- **그러나 도메인 고유어는 별개로 자주 누락·오인식**:

| 용어 | 정답 출현 | STT 반영 | |
|---|---|---|---|
| 노바드림 / GA / CTA / 픽셀 / 세그먼트 / 제안서 | 정상 | 정상 | ✅ |
| A/B | 4회 | 0회 ("AB 돌보") | ❌ |
| 누끼 | 3회 | 0회 ("눕기 컷") | ❌ |
| 채린 | 2회 | 0회 ("체리님") | ❌ |
| 이중 집계 | 1회 | 0회 ("이좀 집계") | ❌ |
| 목업 | 1회 | 0회 ("모겁") | ❌ |

→ **"전체 정확도는 높아도(94%) 비즈니스 핵심 용어에서 집중적으로 틀린다"**가 이 도메인 STT의 특징.
즉 단순 WER만 보면 안 되고, **도메인 용어 정확도를 따로 봐야 한다.**

### 그래서 다운스트림은? — LLM이 흡수

이 노이즈 transcript를 그대로 액션아이템 추출에 넣어도, **LLM이 문맥으로 보정**한다.
예: "눕기 컷"→누끼 컷, "체리님"→채린, "AB 돌보"→A/B 로 이해해 **액션아이템·회의록은 깨끗하게** 생성.
raw transcript는 추적용으로 노이즈째 보존(`note: pure Whisper transcription`).
→ 본 시스템에서 STT 노이즈의 영향이 최종 산출물까지 전파되지 않는 이유.
(실행: `make stt` → `python -m meeting_ai.pipeline data/stt/<파일>.json`)

## 결론 — 성능 좋은 STT는 토대(상한선)다

- 받아쓰기 품질이 **전체 파이프라인 품질의 상한**이다. STT가 "채널별 성과 → 채널된 손가락"으로 망가지면
  뒤의 LLM이 아무리 좋아도 원문 의미를 복원하기 어렵다.
- 특히 **한국어 + 광고 전문용어/약어(GA·CPM·ROAS 등)**에서 소형 모델 오류율이 높다.
- 따라서 실서비스에선 **고성능 STT가 필수**다:
  - 큰 Whisper(`medium`/`large-v3`) 또는 도메인 파인튜닝 모델
  - 광고 용어 사전 기반 후처리(현재 약어 사전과 연계)
  - 외부 유출 금지 제약 하에서는 **온프레미스 고성능 STT** 도입

## 본 시스템의 완충 장치 (STT 오류를 견디는 설계)

STT가 완벽하지 않다는 전제 하에, 아래 계층이 오류 영향을 줄인다:
1. **모델 크기 상향** — 가장 효과적(small→medium에서 GA·성과·제안서 교정). 비용/속도 trade-off.
2. **LLM의 문맥 보정(핵심)** — transcript는 어차피 LLM(추출·요약)을 거친다. LLM이 문맥으로
   "눕기 컷→누끼 컷", "체리님→채린"을 이해해 **최종 구조화 출력(액션아이템·회의록)은 깨끗하게** 나온다.
   raw transcript는 추적용으로 노이즈째 보존. → **STT 노이즈를 STT에서 다 잡을 필요가 없다.**
3. **약어 사전 주입** — LLM이 약어를 오해하지 않도록 컨텍스트 제공
4. **confidence + 사람 검수** — 낮은 신뢰 항목을 대시보드에서 최종 교정
- (참고) initial_prompt 어휘 바이어싱은 본 실험에서 효과가 거의 없어, 핵심 완충책에서 제외.

> 요약: **"모델 크기로 토대를 올리고, 남는 노이즈는 LLM의 문맥 보정 + 사람 검수로 흡수한다.
> initial_prompt 바이어싱은 기대만큼 효과적이지 않았다(실측)."**
