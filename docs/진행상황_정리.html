0. 전체 아키텍처 (한눈에)

음성 mp3→ Whisper STT → transcript JSON→ 정제·청크→ DuckDB 적재→ LLM 추출→ 검증·재시도·보정→ action_items →Slack 페이로드 &Streamlit 대시보드
왜 이 구조? — 모듈을 층으로 분리(수집·처리·추출·표현)해서, LLM 모델을 바꾸거나 스키마를 고쳐도 다른 층에 영향이 안 가게. provider 추상화로 mock/Gemini/Ollama를 환경변수 하나로 토글.

3.2 데이터 파이프라인

단계한 일파일
입력transcript JSON 로드(두 포맷 흡수: 제공 데이터 + 일반 STT)transcript_loader.py
처리화자 정규화 · 약어 사전 · 머뭇거림/중복 제거 · 의미단위 청크preprocess.py
출력DuckDB 적재 + 멱등성(meeting_id 단위 delete-replace)db.py
왜 DuckDB? — 파일 1개 임베디드 OLAP라 대시보드 집계에 강함. 100명 PoC엔 Postgres가 과함, SQLite보다 분석에 우수.
왜 멱등성? — 파이프라인은 현실에서 여러 번 재실행됨(에러 후 재시도 등). 중복 적재·데이터 깨짐 방지. (과제 명시 제약)
완료

3.3 LLM 액션아이템 추출 (평가 핵심)

요구어떻게
프롬프트 설계role 지정 + 도메인 컨텍스트 + few-shot 예시 (prompts.py)
도메인 처리약어 사전 주입(회의별 동적) + 암묵 R&R 매핑("제가 챙길게요"→화자)
구조화 출력JSON 강제(response_mime_type/format) + pydantic 재검증 (이중 방어)
신뢰도 신호confidence(0~1) + source_seg_ids/quote(근거 인용)
검증·재시도스키마 위반 시 강화 프롬프트 재시도 + 환각 필터 + 타입 보정
산출물회의별 액션아이템 + Slack 페이로드 JSON
왜 이렇게? — LLM 출력은 매번 흔들려 못 믿음. "거부만" 하면 데이터 손실, "다 받으면" 깨진 데이터. 그래서 재시도(형식 깨짐)·환각필터(근거 지어냄)·타입보정(사소한 형식차)로 살릴 건 살리고 거를 건 거름. = "신뢰 불가 출력을 신뢰 가능한 자산으로" (평가 포인트).
완료

3.4 분석 대시보드 (위젯 4)

  • ① 주차별 추이(막대=건수, 선=완료율, 최근 12주 고정축)
  • ② 담당자별 미완료 Top N
  • ③ 반복 이슈 키워드 (BoW: 토큰화+조사제거+불용어)
  • ④ confidence 분포 + 낮은 항목 드릴다운
  • + 액션아이템 상태 편집 UI(대기/진행중/완료/지연) · 위치: dashboard/app.py
왜? — "차트 나열"이 아니라 의사결정 흐름: 워크로드 추세→병목 담당자→반복 과제→검수 대상. BoW를 택한 이유는 회의 수 적은 PoC에서 임베딩 클러스터링보다 안정·설명가능.

3. 핵심 설계 결정 & 이유 (면접 대비)

결정왜
추출/트래킹 레이어 분리
(action_items ↔ action_status)		 LLM 재추출 시 제목이 흔들림 → 회의(meeting_id)를 멱등 단위로 추출은 통째 교체하되, 사람이 바꾼 상태는 안정키(action_key)로 별도 보존. "추출은 멱등 재생성 + 상태는 생존" 동시 달성.
provider 추상화 (mock/gemini/ollama) 외부 유출 금지·rate limit·시연 안정성을 모두 만족. 모델 교체가 다른 코드에 안 번짐.
confidence 기반 검수 전수 검수는 비효율. 낮은 confidence(<0.6)만 사람이 보면 검수범위 ~70% 축소.
429 백오프 + 빈 입력 가드 무료 티어 분당 제한·0건 회의에서도 파이프라인이 죽지 않게(견고성).
타입 보정(coercion) 소형 모델이 인용을 리스트로 주는 등 형식 변형 → 거부 대신 변환해 복구율↑ (F1 0.67→0.81 실증).
